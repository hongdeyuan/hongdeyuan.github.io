setState 是异步还是同步，setState 之后发生了哪些事情

发表于 2023-08-15 本文字数 883 字 阅读时长 2 分钟
  • De Hong
首页 React
  • setState 的 “异步” 并不是说内部由异步代码实现，其实本身执行的过程和代码都是同步的，只是合成事件和钩子函数的调用顺序在更新之前，导致在合成事件和钩子函数中没法立马拿到更新后的值，形式了所谓的 “异步”，当然可以通过第二个参数 setState (partialState, callback) 中的 callback 拿到更新后的结果。
  • setState 的批量更新优化也是建立在 “异步”（合成事件、钩子函数）之上的，在原生事件和 setTimeout 中不会批量更新，在 “异步” 中如果对同一个值进行多次 setState ， setState 的批量更新策略会对其进行覆盖，取最后一次的执行，如果是同时 setState 多个不同的值，在更新时会对其进行合并批量更新。
async getList() {
  let res = await axios.get('post/list');
  this.setState({list: res.result});
  console.log(this.state.list);// 此时是同步的，在非时间函数是同步的
}

add () {
  this.setState({count: this.state.count + 1});
  console.lgo(this.state.count); //  此时拿到的是 更新之前的值，在事件函数是异步的
}
  • 总结：
    1. 在一个事件绑定的函数，不管执行多少次 setState，他们也会在函数执行结束之后，被归结为一次重新渲染（用以优化性能），这个等到最后一起执行的行为成为：“批量更新”
    2. React 事件函数使用，像这样使用形式绑定 this.xxx = this.xxx.bind (this) ; 这里批量操作会发生异步表现；是因为 React 知道什么时候退出该事件，什么时候进行批量操作（Batch update）
    3. 其他情况如上面的，获取 post/list 情况，或这样使用 addEventListener 绑定，函数挣脱了 React 的控制， React 不知道什么时候执行批量操作，setState 就会是同步的；
